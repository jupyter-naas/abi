# EventService

A durable, typed event log with live pub/sub broadcast.

Every published event is:

1. **Persisted** to a SQLite-backed append-only log (replayable forever via `query()`).
2. **Broadcast** on the bus topic derived from the event class (delivered live to every connected subscriber via `subscribe()`).

The two paths are independent — if the bus is down, the event still lands in the log; if no one was subscribed at publish time, the event still survives and can be read later. Use the log for catch-up and history, the bus for live reactions.

## Anatomy of an event

Every event is a Pydantic class that inherits from `LogProcess`. `LogProcess` itself is generated by `onto2py` from [EventOntology.ttl](ontologies/modules/EventOntology.ttl) and lives in [ontologies/modules/EventOntology.py](ontologies/modules/EventOntology.py).

The contract `EventService` cares about:

- `_class_uri: ClassVar[str]` — IRI of the class. Used as the routing key on the bus topic and as the event_type in the log. Must be unique per class.
- `_property_uris: ClassVar[dict]` — field-name → predicate-IRI map. Snapshotted into the JSON payload so the schema survives Python-side renames.
- Inherits from the **canonical** `LogProcess` (the one defined in `naas_abi_core.services.event.ontologies.modules.EventOntology`). This is non-negotiable — `EventService.publish` does an `isinstance(event, LogProcess)` check, and `query()` walks `LogProcess.__subclasses__()` to reconstruct typed instances without a `hint_class`.

## Creating a new event type

### 1. Write a TTL

Create a `.ttl` file in your domain's `ontologies/modules/` directory. Three things matter:

1. `owl:imports <http://ontology.naas.ai/abi/EventService>` so `LogProcess` is pulled from the canonical EventOntology (not regenerated locally).
2. `rdfs:subClassOf abi:LogProcess` on each event class.
3. The `abi:pythonPackage` / `abi:ontologyResource` / `abi:pythonResource` locator annotations on your own `owl:Ontology` declaration, so any *other* module can in turn `owl:imports` your ontology by its canonical IRI.

```ttl
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .
@prefix skos: <http://www.w3.org/2004/02/skos/core#> .
@prefix dc: <http://purl.org/dc/terms/> .
@prefix abi: <http://ontology.naas.ai/abi/> .
@prefix mydomain: <http://ontology.naas.ai/abi/mydomain/> .

mydomain:MyDomainEvents a owl:Ontology ;
  dc:title "My domain events"@en ;
  owl:imports <http://ontology.naas.ai/abi/EventService> ;
  abi:pythonPackage    "my_module" ;
  abi:ontologyResource "ontologies/modules/MyDomainEvents.ttl" ;
  abi:pythonResource   "ontologies/modules/MyDomainEvents.py" .

mydomain:UserRegistered a owl:Class ;
    rdfs:label "UserRegistered"@en ;
    rdfs:subClassOf abi:LogProcess ;
    skos:definition "Emitted after a new user account is created."@en .

mydomain:email a owl:DatatypeProperty ;
    rdfs:label "email"@en ;
    rdfs:domain mydomain:UserRegistered ;
    rdfs:range xsd:string .

mydomain:source a owl:DatatypeProperty ;
    rdfs:label "source"@en ;
    rdfs:domain mydomain:UserRegistered ;
    rdfs:range xsd:string .
```

**Why `<http://ontology.naas.ai/abi/EventService>`?** That's the canonical IRI of the EventOntology — a normal, spec-compliant `owl:imports` target. onto2py keeps an index of every ontology declared by an installed `naas_abi*` package; when an `owl:imports` IRI matches one in the index, the TTL is loaded through `importlib.resources` (via the `abi:pythonPackage` / `abi:ontologyResource` annotations on that ontology), so it works the same in editable installs, wheels, and zipped packages. Plain relative paths (e.g. `<../other/Foo.ttl>`) also work for intra-package references. Remote HTTP IRIs that no installed package declares (BFO, etc.) are skipped — they're RDF-level references, not codegen inputs.

### 2. Generate the Python module

```bash
uv run python -m naas_abi_core.utils.onto2py path/to/MyDomainEvents.ttl
```

This produces `path/to/MyDomainEvents.py` next to the TTL (same stem). The generated file:

- Imports `LogProcess` from the canonical EventOntology module (not a regenerated copy).
- Emits one class per `owl:Class` in your TTL, inheriting from `LogProcess`.
- Bakes the predicate-IRI map into each class's `_property_uris`.

You don't edit this file by hand — regenerate when the TTL changes. The generated file carries a sha256 of the inputs at the top so re-running onto2py is a no-op when nothing changed.

### 3. Wire the service into your module

Make sure your `BaseModule.dependencies` lists `EventService`:

```python
from naas_abi_core.services.event.EventService import EventService

class MyModule(BaseModule):
    dependencies = ModuleDependencies(
        services=[EventService, ...],
    )
```

You also get it transitively if you depend on any service that already depends on `EventService` (e.g. `ObjectStorageService`).

## Publishing

```python
from naas_abi_core.module.Module import BaseModule
from path.to.MyDomainEvents import UserRegistered

class MyModule(BaseModule):
    def some_action(self, email: str, source: str) -> None:
        evt = UserRegistered(email=email, source=source)
        self.engine.services.events.publish(evt)
```

What `publish` does:

1. `isinstance(event, LogProcess)` check — raises `InvalidEventError` otherwise.
2. Populates `created_at` if you didn't.
3. Serializes via `EventCodec.serialize` (Pydantic → JSON dict + the predicate-IRI snapshot).
4. Appends to the SQLite log (`EventSQLiteAdapter.append`).
5. Broadcasts on `bus.publish(class_iri_to_topic(_class_uri), event._uri, payload)`.

If the bus is unavailable or fails, the persisted log is still authoritative — you'll see a warning but `publish` won't raise. Durability over liveness.

## Subscribing (live events)

Every call to `subscribe` registers an **independent** listener. Two subscribers on the same event class both receive every event — they don't compete.

```python
def on_user_registered(evt: UserRegistered) -> None:
    print(f"Welcome {evt.email}")

self.engine.services.events.subscribe(UserRegistered, on_user_registered)
```

### With a content filter

Filters are evaluated in-memory against the JSON payload before the (more expensive) Pydantic reconstruction. Same EventBridge-style dict syntax as `query()`.

```python
# Only Gmail users
self.engine.services.events.subscribe(
    UserRegistered,
    on_user_registered,
    filter={"email": {"suffix": "@gmail.com"}},
)

# Only OAuth-sourced registrations
self.engine.services.events.subscribe(
    UserRegistered,
    on_user_registered,
    filter={"source": "oauth"},
)

# Equality, suffix/prefix/contains, in, gt/gte/lt/lte — see EventFilter for the full set.
```

**Live-only**: subscribers don't replay history. If you need catch-up, use `query_for_consumer()` below.

## Querying (history, replay)

```python
events = self.engine.services.events.query(event_class=UserRegistered)
# → [UserRegistered, UserRegistered, ...]
```

Filters, time/seq bounds, and limits supported:

```python
events = self.engine.services.events.query(
    event_class=UserRegistered,
    since_timestamp="2026-01-01T00:00:00",
    filter={"source": {"in": ["oauth", "magic_link"]}},
    limit=100,
)
```

For large windows use `iter_query()` — streams in `batch_size` chunks under snapshot semantics (max `seq` captured at first call, no events appended during iteration leak in):

```python
for evt in self.engine.services.events.iter_query(event_class=UserRegistered, batch_size=500):
    process(evt)
```

### Untyped queries (no `event_class`)

```python
events = self.engine.services.events.query()
```

Returns instances of whatever subclass each event was published as — resolved via `LogProcess.__subclasses__()`. This is **only correct** when the publishing class was generated with `owl:imports` (so it inherits from the canonical `LogProcess`). Classes from regenerated-self-contained modules would deserialize as bare `LogProcess` and lose their typed fields. The `owl:imports` step in §1 is what makes this work.

## Consumer-cursor queries (durable catch-up)

For a worker that needs to "process every event I haven't seen yet" across restarts:

```python
events = self.engine.services.events.query_for_consumer(
    consumer_id="my-worker",
    event_class=UserRegistered,
    limit=100,
)
# cursor advances atomically — next call returns events strictly after the last batch
```

`iter_query_for_consumer()` drains in batches with the same atomic-cursor-advance semantics.

## Filter DSL cheatsheet

The same dict syntax works for `query(..., filter=...)` and `subscribe(..., filter=...)`. Operators are evaluated as a nested map.

| Form | Meaning |
|---|---|
| `{"field": "value"}` | Equality |
| `{"field": {"prefix": "abc"}}` | starts with `abc` |
| `{"field": {"suffix": ".pdf"}}` | ends with `.pdf` |
| `{"field": {"contains": "foo"}}` | contains substring |
| `{"field": {"in": ["a", "b"]}}` | one of |
| `{"field": {"gt": 5}}` | `> 5` (also `gte`, `lt`, `lte`) |

Multiple keys in the filter are AND-ed. See [EventFilter.py](EventFilter.py) for the full evaluator and SQL pushdown.

## Common pitfalls

- **Forgetting `owl:imports`** — onto2py will silently regenerate its own copy of `LogProcess`, and your subclass won't pass `isinstance(event, LogProcess)` in `publish()`. Symptom: `InvalidEventError` despite the class "being" a LogProcess by name.
- **Naming the generated file differently from the TTL** — the resolver assumes `Foo.ttl` → `Foo.py` in the same directory. Don't pass a custom output filename to `onto2py`.
- **Expecting `subscribe()` to replay history** — it doesn't. Use `query()` or `query_for_consumer()`.
- **Heavy work in a subscriber callback** — callbacks run on the bus consumer thread. For long-running work, hand off to a queue.
