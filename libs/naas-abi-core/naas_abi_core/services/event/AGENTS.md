# Event Service — AGENTS.md

> Scope: `libs/naas-abi-core/naas_abi_core/services/event/`. Canonical reference for agents. Also see `README.md` for tutorials.

## Purpose

Durable, typed event log with live pub/sub fanout. Two independent paths:

- **Log** — every published event is persisted append-only (replayable forever).
- **Bus** — same event broadcast on the bus for live subscribers.

Subscribers do **not** automatically replay history; for catch-up use `query` / `query_for_consumer`.

## Port (`EventPort.py`)

### `IEventAdapter` (secondary)

```python
append(event_id, event_type, timestamp, payload: bytes) -> StoredEvent
query(event_type, since_seq, until_seq,
      since_timestamp, until_timestamp,
      json_filter, limit) -> list[StoredEvent]
max_seq(event_type=None) -> int
get_cursor(consumer_id, event_type) -> int
query_for_consumer(consumer_id, event_type, limit) -> list[StoredEvent]   # atomic cursor advance
```

## Service API (`EventService.py`)

```python
publish(event) -> StoredEvent
# event must be an instance of a LogProcess subclass; otherwise InvalidEventError.

query(event_class=None, since_seq=None, until_seq=None,
      since_timestamp=None, until_timestamp=None,
      filter=None, limit=None) -> list[Any]

iter_query(event_class, since_seq=None, since_timestamp=None,
           until_timestamp=None, filter=None, limit=None,
           batch_size=500) -> Iterator[Any]
# Snapshot semantics: max seq captured at first call — no leak-in of later appends.

query_for_consumer(consumer_id, event_class, limit=None) -> list[Any]
iter_query_for_consumer(consumer_id, event_class, limit=None, batch_size=500) -> Iterator[Any]
# Atomically advance the consumer cursor per batch.

subscribe(event_class, callback, filter=None) -> Thread
# Live-only fanout. Each subscriber is independent. Filter evaluated in-memory.
```

## Filter DSL

EventBridge-style: `eq`, `prefix`, `suffix`, `contains`, `in`, `gt`, `gte`, `lt`, `lte`. Supports JSON-path keys. See `EventFilter.py`. The SQLite adapter pushes filters down via `build_where`; the rest is evaluated in-process via `matches`.

## Event Ontology

- Generated by `onto2py` from `ontologies/modules/EventOntology.ttl`.
- Base hierarchy: `RDFEntity` → `Process` → `LogProcess` (canonical event base).
- Every publishable event must inherit `LogProcess` (via `owl:imports`).
- Event metadata carried at publish time:
  - `_class_uri` — IRI used as routing key + `event_type`.
  - `_property_uris` — field-name → predicate-IRI snapshot (frozen at publish).

## Adding a new adapter

1. Implement `IEventAdapter` in `adapters/secondary/<Name>Adapter.py`. All five methods.
2. The payload is opaque bytes — preserve it exactly. JSON metadata (`_uri`, `_class_uri`, `_property_uris`) is handled by `EventCodec`.
3. Support the EventBridge-style filter DSL — at minimum via in-memory `matches`; pushdown to the backend if it has a query language.
4. `query_for_consumer` **must** advance the cursor atomically with the read.

## Adding a new event type

1. Define the class in the appropriate `ontologies/classes/.../<EventName>.py` module, inheriting from the canonical `LogProcess` (transitively).
2. Set `_class_uri` and `_property_uris` on the class.
3. Publish via `event_service.publish(<EventInstance>())`. **Do not** invent ad-hoc event types — they will be rejected with `InvalidEventError`.
